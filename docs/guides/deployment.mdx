The same binary deploys three ways, selected by how you split the `ROLE`.

## All-in-one

`ROLE=all` runs every role in one process with an in-memory bus. Only Postgres
and Redis are required — no Kafka.

```bash
cp .env.example .env   # ROLE=all
docker compose up --build
```

Best for local development, demos, and small instances.

<Warning>
  The book is in memory with no Kafka. It still recovers from the durable Postgres
  checkpoint on restart, but without a durable event log the last few
  un-checkpointed events can't be tail-folded. Use split roles for production.
</Warning>

## Single-VM, split roles

Each role runs as its own container on one host, wired together by Redpanda
(Kafka), Postgres, and Redis.

```bash
docker compose -f docker-compose.prod.yml up -d
```

Required variables are validated at startup (the compose file uses
`${VAR:?message}` so a missing value fails loudly). The engine's gRPC port is
bound to localhost only; put TLS termination and a reverse proxy in front of the
gateway.

## Multi-VM

| Component | Placement | Scaling |
|-----------|-----------|---------|
| Engine | Dedicated host(s) | By markets per instance, or instances per symbol group |
| Gateway | Behind a load balancer with **sticky sessions** (required for WebSocket) | Horizontally |
| Workers | Dedicated host(s) | By Kafka partitions |
| Postgres / Redis / Kafka | Managed services | Independently |

## Building the image

```bash
docker build -t clob-server .
```

The `Dockerfile` is a two-stage build producing a static binary on `alpine`,
exposing `8080` (HTTP) and `50051` (gRPC).

## Health & observability

| Endpoint | Purpose |
|----------|---------|
| `GET /health` | Liveness — `{"status":"ok"}` |
| `GET /metrics` | Prometheus metrics (HTTP latency/counts, orders placed, worker events). Expose only on a trusted network. |

Logs are structured JSON (zerolog) in non-local environments, with a request ID
on every HTTP request.

## Crash recovery (event-sourced)

On restart the engine rebuilds each market's resting book by folding its own
`market-events` log — so **open orders survive a restart**, no mass cancellation.

- The **`booksnapshot`** role continuously folds the log into a per-market
  checkpoint in the `book_snapshots` table, bounding how far recovery has to
  replay. Deploy it alongside the other roles (it is in both compose files).
- On startup the engine loads the latest checkpoint, folds any newer events, and
  seeds the book via the library's `WithInitialOrders` seam (no re-matching, no
  re-run of the credit hook, no re-emitted events). The event sequence resumes
  above the last recovered event so worker idempotency holds. **Reserved credits
  never moved during the crash, so recovery touches no wallets.**

Set `ENGINE_RECOVERY=cancel` to opt back into the old behaviour (cancel open
orders and release reservations on restart). `ROLE=all` uses that fallback
automatically beyond the checkpoint, since it has no durable log to tail.

<Note>
  Recovery is consistent up to the last checkpointed (and tail-folded) event.
  A bit-exact alternative — a command write-ahead log replayed through matching —
  is the documented next step.
</Note>
